# Sessions (세션 체크포인트)

opencode 세션 중단, 작업 인계, 체크포인트를 Markdown으로 기록하는 위치입니다.

세션 파일은 실행 기준이 아니라 인계 체크포인트입니다. 공식 요구사항, 계약 경계, 계획을 대체하지 않습니다.

## Naming

세션 파일은 다음 형식을 사용합니다.

```text
YYYY-MM-DD_HH-MM-SS_short-slug.md
```

예시:

```text
2026-06-06_11-16-55_oh-my-opencode-slim-experiment.md
```

## Usage

- 새 세션이나 장기 작업 시작 시 가장 최근 세션 파일 1개를 먼저 확인합니다.
- 과거 세션은 사용자가 명시하거나 에이전트가 참조 이유를 설명해 승인받은 경우에만 검색해서 읽습니다.
- 세션 파일은 대화 전문이 아니라 체크포인트와 goal 리스트로 사용합니다.
- 큰 작업 단위 종료, 하루 단위 종료, 재시작 전, 방향 전환 전후에 저장합니다.
- 장기 작업 시작, 계획 승인 후, 구현 상태가 크게 바뀐 뒤, 열린 질문이나 블로커가 생긴 뒤에는 세션 작성 또는 갱신을 검토합니다.
- 세션에 기록한 내용이 계속 필요한 운영 기준이면 `README.md`, `PROJECT.md`, `AGENTS.md`, `docs/`로 승격합니다.

## Update Policy (업데이트 정책)

세션 체크포인트는 모든 턴마다 작성하지 않습니다. 다음 작업자가 이어받을 때 필요한 상태가 생겼을 때 작성하거나 갱신합니다.

MUST:

- 장기 작업을 중단하기 직전
- 하루 단위 작업 종료 시점
- 큰 작업 단위가 끝났을 때
- 열린 질문, 블로커, 사용자 결정 대기가 생겼을 때
- 다음 세션에서 이어서 해야 할 `Next Steps`가 명확할 때

SHOULD:

- 계획서가 `doing` 또는 `archived`로 이동했을 때
- 여러 파일을 수정했고 아직 커밋하지 않았을 때
- 검증 실패나 미완료 후속 작업이 남았을 때
- 작업이 여러 턴에 걸쳐 복잡해졌을 때

MAY SKIP:

- 단일 파일의 작은 수정
- 바로 완료되고 커밋된 작은 작업
- 의견 제시나 단순 질의응답
- 공식 문서, 계획서, 커밋 메시지만으로 상태가 충분히 복원되는 작업

세션은 짧게 유지합니다. 대화 전문이 아니라 목표, 현재 상태, 완료 작업, 다음 단계, 열린 질문, 중요 파일, 실행 명령, 리스크 중심으로 적습니다.

## Template

```md
# Session Checkpoint (세션 체크포인트)

## Goal (목표)

이번 세션 또는 작업 단위의 목표를 적습니다.

## Current Status (현재 상태)

지금 어디까지 진행됐고, 무엇이 남았는지 적습니다.

## Completed (완료한 작업)

완료한 변경, 검증, 결정 사항을 적습니다.

## Next Steps (다음 단계)

다음 작업자가 바로 이어서 할 일을 적습니다.

## Open Questions (열린 질문)

사용자 확인이 필요하거나 아직 정하지 못한 질문을 적습니다.

## Important Files (중요 파일)

다음 작업자가 먼저 확인해야 할 파일 경로를 적습니다.

## Commands Run (실행한 명령)

중요한 검증, 빌드, 테스트 명령과 결과를 적습니다.

## Risks / Notes (리스크/메모)

주의해야 할 리스크, 미해결 이슈, 관련 discussion, plan, contract, architecture 경로를 적습니다.
```
