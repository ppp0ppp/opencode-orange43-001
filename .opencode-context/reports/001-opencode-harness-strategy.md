# opencode 기본 설정 템플릿 구축 보고서

작성일: 2026-06-06

## 결론

`oh-my-opencode-slim` 같은 준비된 하네스 구조는 가져와서 시작하는 방향이 합리적입니다. 다만 그대로 전면 도입하기보다, 현재 프로젝트를 "개인/팀 기본 템플릿"으로 보고 작은 단위로 벤더링하거나 포크한 뒤 검증된 구성만 흡수하는 방식이 안전합니다.

핵심 방향은 다음과 같습니다.

- 외부 하네스는 초기 가속 장치로 사용하되, 권한/에이전트/스킬/플러그인은 로컬 정책으로 통제합니다.
- 스키마와 계약 경계면은 별도 문서로 고정하고, 변경 시 반드시 사용자 확인을 요구하는 규칙을 둡니다.
- 세션 유실 문제는 opencode 세션 자체를 대체하기보다, 수동 또는 명령 기반의 Markdown 체크포인트로 보완합니다.
- 장기 컨텍스트 교환용 디렉토리는 `docs/`와 분리해 `.opencode-context/`를 사용합니다.
- 모델 변경 스트레스는 하네스 자체를 모델별로 갈아엎는 방식보다, 계약/평가/권한/체크리스트를 안정화해 줄이는 편이 좋습니다.

## 전제와 미확인 사항

이 보고서는 현재 대화와 로컬 설정 상태를 기준으로 작성했습니다. `oh-my-opencode-slim`, `graphify`, `tidy-tdd-first`, `github` skill의 실제 라이선스, opencode 버전 호환성, 보안 모델은 추가 검증이 필요합니다.

실제 도입 전에는 다음을 확인해야 합니다.

- 정확한 저장소 URL 또는 패키지 이름
- 라이선스와 유지보수 상태
- opencode 현재 버전과의 설정 스키마 호환성
- 플러그인/스킬이 실행하는 도구와 권한 범위
- 외부 네트워크, GitHub 토큰, 로컬 파일 접근 여부

## 1. oh-my-opencode-slim 같은 하네스 사용에 대한 의견

준비된 하네스를 사용하는 것은 좋습니다. 직접 codemap, grep, websearch, supervisor agent, subagent, 스킬 체계, 커맨드, 권한 규칙을 모두 설계하려면 시간이 많이 들고, 초기에 잘못 설계한 구조가 오래 남을 수 있습니다.

다만 전면 복붙은 피하는 것이 좋습니다. 추천 방식은 다음입니다.

- 1단계: 외부 하네스를 별도 브랜치나 임시 디렉토리에서 분석합니다.
- 2단계: 기능을 `agent`, `skill`, `plugin`, `command`, `permission`, `workflow` 단위로 분해합니다.
- 3단계: 이 템플릿에 필요한 것만 작게 가져옵니다.
- 4단계: 가져온 항목마다 목적, 트리거, 권한, 실패 시 동작을 문서화합니다.
- 5단계: opencode 업데이트나 모델 변경 시 비교 가능한 기준 파일을 유지합니다.

가져올 때 우선순위는 `읽기/분석 보조`가 가장 높고, `자동 편집/자동 실행`은 가장 낮게 두는 것이 안전합니다.

## 2. tiddy-tdd-first, github 스킬 추가

스킬 추가 자체는 적절합니다. 특히 TDD 흐름과 GitHub 흐름은 자주 반복되는 작업이므로 skill로 분리할 가치가 있습니다.

단, 스킬은 너무 넓게 발동되면 오히려 모델 행동을 흐립니다. description에서 트리거 조건을 좁게 잡는 것이 중요합니다.

추천 설계는 다음과 같습니다.

- `tiddy-tdd-first`: 테스트 우선 구현, 회귀 재현, 실패 테스트 작성, 최소 구현, 리팩터링 순서를 강제하는 스킬
- `github`: issue, PR, review, check, release, gh CLI 사용 흐름을 다루는 스킬
- 두 스킬 모두 파일 편집이나 GitHub 변경 작업을 하기 전 현재 변경사항과 권한을 확인하도록 지시
- `github` 스킬은 토큰, 원격 저장소, PR 생성/머지 같은 외부 부작용이 있으므로 기본적으로 확인을 요구

주의할 점은 `tiddy-tdd-first`라는 이름이 의도한 철자가 맞는지 확인이 필요하다는 것입니다. 일반적으로는 `tidy-tdd-first` 또는 `tdd-first`가 더 자연스럽습니다. 다만 이미 사용 중인 이름이나 외부 skill 이름이 `tiddy-tdd-first`라면 그대로 유지하는 편이 좋습니다.

## 3. graphify 플러그인 추가

`graphify`는 코드 구조, 의존성, 작업 흐름, 세션 맥락을 그래프로 시각화하거나 요약하는 용도라면 이 템플릿과 잘 맞습니다.

도입 전 검토해야 할 리스크는 다음입니다.

- 프로젝트 파일 전체를 스캔하는지 여부
- 외부 서비스로 데이터를 전송하는지 여부
- 생성물이 저장소에 남는지 여부
- opencode plugin API 버전과 호환되는지 여부
- 플러그인이 tool 실행 전후를 가로채며 권한 모델을 우회하지 않는지 여부

추천 도입 방식은 바로 `opencode.json`에 추가하기보다, 먼저 `.opencode/plugin/` 또는 별도 브랜치에서 시험하는 것입니다. 출력물이 유용하고 권한 경계가 명확하면 로컬 설정에 편입합니다.

## 4. 세션 유실 대비 파일 저장

세션을 파일로 보완하는 것은 필요합니다. 특히 터미널에서 `Ctrl+C`로 세션이 종료되는 문제는 실제 사용성을 해칩니다.

다만 모든 대화를 자동 저장하는 방식은 추천하지 않습니다. 노이즈가 많고, 민감 정보가 섞일 수 있으며, 나중에 모델이 읽을 때 오히려 혼란을 줄 수 있습니다.

추천 방식은 `체크포인트 Markdown`입니다.

- 작업 전: 목표, 제약, 현재 상태를 짧게 기록
- 중요한 결정 후: 결정 내용과 이유를 기록
- 중단 전: 다음에 이어갈 작업, 열린 질문, 검증 상태를 기록
- 완료 후: 변경 파일, 검증 결과, 남은 리스크를 기록

향후 opencode command로 `/checkpoint`, `/handoff`, `/decision` 같은 명령을 만드는 것이 좋습니다. 자동화는 이후 플러그인으로 확장하되, 초기에는 명령 기반 수동 저장이 안전합니다.

## 5. 스키마와 계약 경계면 고정

이 항목은 강하게 반영하는 것이 좋습니다. 스키마, API 계약, DB 모델, 이벤트 구조, 모듈 경계는 모델이 임의로 바꾸면 피해가 큽니다.

추천 정책은 다음입니다.

- 계약 경계면은 별도 문서에 명시합니다.
- 계약 변경이 필요한 경우 opencode는 수정 전에 반드시 사용자에게 묻습니다.
- 계약 변경 요청에는 영향 범위, 마이그레이션 필요 여부, 대안, 테스트 계획을 포함해야 합니다.
- 임시 우회, 호환성 레이어, 필드 추가/삭제는 명시 승인 없이는 금지합니다.

이 규칙은 `AGENTS.md`에 넣는 것이 좋습니다. 또한 장기적으로는 `ai-context/contracts/` 아래에 계약 문서를 두고, 중요한 결정은 ADR 형식으로 남기는 것을 추천합니다.

## 6. Markdown 기반 큰 컨텍스트 교환과 디렉토리 이름

Markdown 보고서로 큰 맥락을 주고받는 방식은 적합합니다. 모델과 사용자가 같은 작업 공간을 공유하는 환경에서는 대화보다 파일이 더 안정적인 기준점이 됩니다.

`docs/`는 사용자 문서, 제품 문서, 개발자 문서와 섞일 가능성이 있어 opencode 전용 컨텍스트 저장소로는 애매합니다.

채택 디렉토리명은 `.opencode-context/`입니다.

이유는 다음입니다.

- `docs/`보다 목적이 명확합니다.
- opencode가 참조해야 하는 장기 컨텍스트임이 이름에 드러납니다.
- 일반 제품/개발 문서와 섞이지 않습니다.
- 보고서, 결정 기록, 세션 체크포인트, 계약 문서를 모두 담을 수 있습니다.

추천 구조는 다음입니다.

```text
.opencode-context/
  README.md
  reports/
    001-opencode-harness-strategy.md
  decisions/
    001-contract-boundary-policy.md
  sessions/
    001-initial-template-setup.md
  contracts/
    README.md
```

파일명은 `001-title.md` 형식이 좋습니다. 날짜 기반 이름은 정렬은 쉽지만 주제 파악이 느리고, 순번 기반 이름은 논의의 흐름을 따라가기 좋습니다. 필요하면 `001-2026-06-06-title.md`처럼 혼합할 수 있습니다.

이번 보고서는 `.opencode-context/reports/001-opencode-harness-strategy.md`로 이동했습니다.

## 7. 하네스를 가져와서 수정하는 방향

가져와서 조금씩 수정하는 방향이 현실적입니다. 단, 하네스는 의존성이 아니라 재료로 취급하는 것이 좋습니다.

권장 운영 방식은 다음입니다.

- 외부 하네스 원본을 그대로 추적할 수 있게 출처를 기록합니다.
- 가져온 파일마다 왜 필요한지 한 줄 설명을 둡니다.
- 처음부터 많은 agent를 켜지 않습니다.
- supervisor agent는 권한을 강하게 제한하고, 실제 편집은 build agent나 사용자 확인을 거치게 합니다.
- websearch, github, graph 관련 기능은 외부 네트워크와 토큰 사용 여부를 명시합니다.
- codemap/grep 계열은 읽기 전용 보조로 먼저 도입합니다.

초기 추천 범위는 다음입니다.

- codemap 또는 repository map
- grep/search workflow
- TDD skill
- GitHub skill
- checkpoint/report command
- contract boundary instruction

나중에 추가할 범위는 다음입니다.

- supervisor agent
- graphify plugin
- 자동 세션 요약 plugin
- 외부 websearch 강화

## 8. 모델 변경과 하네스 재설계 스트레스

모델이 바뀔 때 하네스와 에이전트 구조를 다시 짜야 할 수 있다는 우려는 타당합니다. 모델마다 장점, 도구 사용 방식, 과잉 추론, 지시 준수 성향이 달라질 수 있습니다.

이를 줄이려면 하네스를 특정 모델의 성격에 맞추기보다, 모델과 독립적인 안정 장치를 만들어야 합니다.

추천 안정 장치는 다음입니다.

- 계약 경계면 변경 전 확인 규칙
- 파괴적 명령 차단 권한
- 테스트 우선 또는 검증 우선 workflow
- 코드 리뷰 출력 형식 고정
- 보고서/체크포인트 형식 고정
- 하네스 변경 로그
- 모델 업데이트 후 실행할 간단한 평가 시나리오

모델별 차이는 `model profile` 문서로 흡수하는 편이 좋습니다. 예를 들어 특정 모델이 과감하게 리팩터링하는 성향이 강하면 `최소 변경`, `계약 변경 전 질문`, `새 추상화 금지` 같은 규칙을 강화합니다. 반대로 도구 사용이 약한 모델이면 탐색 절차와 검증 명령을 더 명시합니다.

## 권장 다음 단계

1. `ai-context/` 디렉토리 구조를 만들지 결정합니다.
2. `AGENTS.md`에 계약 경계면 변경 전 확인 규칙을 추가합니다.
3. `tiddy-tdd-first` 또는 `tidy-tdd-first` 명칭을 확정합니다.
4. `oh-my-opencode`, `graphify`, `github` skill의 정확한 저장소 URL을 확인합니다.
5. 외부 하네스를 바로 병합하지 말고 분석 보고서를 먼저 작성합니다.
6. `/checkpoint`, `/handoff`, `/decision` command를 추가합니다.
7. 최소 skill 1개와 command 1개를 먼저 추가해 실제 사용감을 검증합니다.

## 지금 시점의 추천 결정안

- 하네스: `oh-my-opencode-slim`을 참고하되, 검증된 부분만 로컬 템플릿에 흡수
- 스킬: TDD와 GitHub는 추가하되 트리거를 좁게 설계
- 플러그인: `graphify`는 보류 후 별도 검증
- 세션 유지: 자동 전체 저장보다 Markdown 체크포인트 우선
- 계약 경계: 사용자 승인 없이는 변경 금지
- 컨텍스트 디렉토리: `.opencode-context/` 사용
- 모델 변경 대응: 하네스 재작성보다 평가/권한/계약 규칙으로 안정화

## 사용자에게 확인할 질문

다음 단계로 넘어가기 전에 아래만 확인하면 됩니다.

1. `tidy-tdd-first`를 별도 skill 이름으로 둘지, 현재 `tdd-workflow` skill을 유지할지 정해야 합니다.
2. `oh-my-opencode-slim`에서 어떤 agent/command/workflow를 먼저 흡수할지 정해야 합니다.
3. `graphify`를 skill로만 쓸지, hook/plugin/MCP까지 활성화할지 정해야 합니다.
4. 현재 `git` skill의 Git Flow 고정 정책을 모든 프로젝트 기본값으로 둘지 정해야 합니다.
